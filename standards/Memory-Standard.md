# Memory Standard v0.2

> Hermes 专家长期记忆的设计规范
>
> **v0.2 更新**：合并与 ChatGPT 第二阶段讨论的新增字段（Priority、Conflict、Decision、Anti-pattern、Checklist、Update Log、来源分级A-E、3问测试、废弃规范）。引入 Memory Atom（MA）概念。

---

## 一、Memory 是什么

Memory 不是"提示词"，不是"角色设定"，不是"运营技巧合集"。

Memory 是**经过人工审核、多源验证、知识蒸馏的专家认知资产**。

它应该像操盘手大脑里默认存在的知识：不需要搜索、不需要推理、天然就知道。

---

## 二、知识原子化体系（v0.2 新增）

本项目采用四级知识粒度：

```
Memory Atom (MA)   → 不可再拆的原子认知（一句话原则）
       ↓ 聚合
Memory (M)         → 一篇完整的专家记忆（由多个 Atom 组成）
       ↓ 被调用
Framework (F)      → 分析框架（调用多个 Memory）
       ↓ 被调用
Skill (S)          → 执行能力（调用 F + M + Checklist）
```

### Memory Atom（MA）

一句不可再拆分的专家认知。例如：

- `MA0001`：封面的唯一目标是提高点击率。
- `MA0002`：一张封面同时只能表达一个信息点。
- `MA0003`：前3秒唯一任务是阻止用户划走。

多个 MA 聚合后形成一篇 M。MA 是知识的最小可复用单元。

详见：[Memory-Atom-Standard.md](Memory-Atom-Standard.md)

---

## 三、Memory 编号规范

统一格式：

```text
M0001
M0002
M0003
...
```

- 4位数字，零填充
- 编号**永久锁定**，删除后不复用
- 废弃的 Memory 标记 `Status: Deprecated`，不删除，保留历史

MA 编号格式：

```text
MA0001
MA0002
...
```

## 四、Memory 分级体系（L1-L4）

| 等级 | 名称 | 内容 | 审核频率 | 锁定期 |
|------|------|------|----------|--------|
| **L1** | 世界观 | 为什么做内容、创作哲学、用户本质 | 每年审核 | 永久 |
| **L2** | 战略 | 推荐机制、IP成长、用户心理、商业模式 | 每半年审核 | 12 个月 |
| **L3** | 战术 | 昵称、简介、封面、标题、脚本、剪辑 | 每季度审核 | 6 个月 |
| **L4** | 平台规则 | 算法机制、流量政策、新功能 | 平台更新立刻审核 | 3 个月 |

## 五、Memory 完整字段规范

### 5.1 Metadata（元数据）

```yaml
ID: M0001
Title: {标题}
Category: {分类}
Level: {L1 | L2 | L3 | L4}
Version: 1.0
Status: {Active | Draft | Deprecated}
Priority: {Critical | High | Normal | Low}
Created: YYYY-MM-DD
Updated: YYYY-MM-DD
Author: {创建者}
```

### 5.2 Summary（一句话概述）

20-50字。AI 快速理解的摘要。必须直接、精确。

```markdown
## Summary

{一句话表达整篇 Memory 最核心的认知。像 CPU 指令一样直接。}
```

### 5.3 Core Principle（核心原则）

只能有一个核心原则。不要写多个。

```markdown
## Core Principle

{这条 Memory 唯一的核心原则。}
```

### 5.4 Why（为什么成立）

解释底层逻辑。不能只是经验复述。

```markdown
## Why

{解释底层机制。不是"怎么做"，而是"为什么这样做有效"。}
```

### 5.5 How（如何执行）

必须有操作锚点。不能空谈。

```markdown
## How

{具体操作步骤。每一步必须能指导实际动作。}
```

### 5.6 Decision（决策规则）

AI 什么时候调用这条 Memory。

```markdown
## Decision

- 触发场景：{什么时候调用}
- 输入要求：{需要什么信息才能判断}
- 输出：{调用后会产出什么}
```

### 5.7 Anti-pattern（反模式）

明确指出**不要做什么**。

```markdown
## Anti-pattern

- ❌ {错误的做法}
- ❌ {另一种错误做法}
```

### 5.8 Evidence（证据）

不仅标注来源，还要说明**为什么相信**。

```markdown
## Evidence

### 理论支撑
{底层的理论或机制解释}

### 案例验证
{实际案例中的表现}

### 行业共识
{多位操盘手的共识程度}
```

### 5.9 Confidence（可信度）

| 星级 | 含义 | 条件 |
|------|------|------|
| ★★★★★ | 行业底层规律 | 官方资料 + 多位优秀操盘手 + 长期验证 + 多案例 |
| ★★★★☆ | 高度可信 | 至少两个高质量来源 + 大量案例 |
| ★★★☆☆ | 可信 | 来源可信但案例较少 |
| ★★☆☆☆ | 待验证 | 少量经验 |
| ★☆☆☆☆ | 假设 | 暂时保留，等待验证 |

> Memory 最低要求 **★★★☆☆**。
> ★★☆☆☆ 及以下只留在 `knowledge/`，不进入 `memory/`。

### 5.10 Sources（来源）

采用 **A-E 五级来源制**：

| 等级 | 类型 | 示例 |
|------|------|------|
| **A** | 官方 | 抖音平台文档、白皮书、官方公开课 |
| **B** | 成熟机构 | MCN 培训资料、行业报告、长期操盘手 |
| **C** | 优秀个体 | 知名创作者、课程、采访、博客 |
| **D** | 社区经验 | 社区精华、论坛观点 |
| **E** | 未验证 | **默认不进入 Memory** |

每条 Memory 至少需要 **2 个 B 级以上来源** 或 **3 个 C 级以上来源**。

```markdown
## Sources

- [A] 抖音创作者学习中心 — 《推荐算法白皮书》2025
- [B] 李佳琦团队 — 《短视频运营手册》第3章
- [B] 池骋 — 《起号方法论》课程第5节
```

### 5.11 Scope（适用范围）

```markdown
## Scope

### 适用于
- 知识IP
- 旅行IP
- 风景账号
- 剧情号
- 新人账号

### 不适用于
- 直播切片
- 娱乐号
- 影视混剪
```

### 5.12 Conflict（冲突项）

记录与其他 Memory 的已知冲突，帮助 AI 权衡。

```markdown
## Conflict

- 与 [M0032](../L3_Tactics/M0032.md) 存在部分冲突：{冲突说明}
- 当两者冲突时，以 {优先级规则} 为准。
```

### 5.13 Related Memory（关联记忆）

```markdown
## Related Memory

- [M0003](../L2_Strategy/M0003.md) — 起号策略
- [M0018](../L3_Tactics/M0018.md) — 封面设计
- [MA0001](../../atoms/MA0001.md) — 封面唯一目标
```

### 5.14 Checklist（内嵌检查项）

```markdown
## Checklist

- [ ] 是否只有一个核心观点？
- [ ] 是否建立了点击理由？
- [ ] 是否符合长期 IP？
- [ ] 是否违背创作哲学？
```

### 5.15 Update Log（更新记录）

```markdown
## Update Log

| 版本 | 日期 | 变更内容 |
|------|------|----------|
| v1.0 | 2026-06-27 | 首次建立 |
| v1.1 | 2026-07-15 | 增加适用范围 |

```

## 六、收录标准（6 条铁律 + 3 问测试）

### 6 条铁律（必须全部满足）

1. ✅ **操作锚点**：能指导至少一个具体动作
2. ✅ **解释为什么**：揭示底层机制
3. ✅ **可验证**：有 ≥3 个独立来源或 ≥2 个 B 级以上来源
4. ✅ **可长期**：跨时间复用（非短期热点）
5. ✅ **抖音相关**：与抖音生态强相关
6. ✅ **不违哲学**：不违背创作者哲学（长期 IP > 短期流量）

### 3 问测试（v0.2 新增）

每条 Memory 必须能回答三个问题：

1. **为什么？** — 这个原则为什么成立？（底层机制）
2. **什么时候？** — 什么时候该用它？什么时候不该？
3. **为什么不是其它方法？** — 它比替代方案好在哪里？

> 如果回答不了任何一个问题，说明它还不是合格 Memory。

### ❌ 禁止收录

| 类型 | 示例 | 问题 |
|------|------|------|
| 废话 | "做好定位"、"坚持更新" | 无操作锚点 |
| 个人经验 | "我这样做就爆了" | 未多源验证 |
| 平台规则复述 | "抖音推荐靠算法" | 没有解释为什么 |
| 热点技巧 | "2026年某流量扶持政策" | 不可跨时间复用 |
| 创作者哲学冲突 | "标题一定要制造焦虑" | 损害长期 IP |

## 七、废弃规范

Memory 不删除。改为：

```yaml
Status: Deprecated
```

并在 Update Log 中记录废弃原因。保留所有历史版本，方便 AI 理解知识演化过程。

---

## 八、审核流程

```
提议新 Memory
    ↓
检查是否满足 6 条铁律 ← 不满足 → 拒绝或留在 knowledge/
    ↓
通过 3 问测试          ← 不通过 → 修改后重新提交
    ↓
多源验证（来源 A-E 分级）← 不足 → 标记为待验证
    ↓
人工审核               ← 不通过 → 修改后重新提交
    ↓
分配等级 + 编号
    ↓
进入 memory/ 对应目录
    ↓
记录到 config/memory-index.md
```
