# Memory Standard.md

# Hermes Expert Memory Standard

Version: 0.1

---

# 一、设计目标

Hermes Memory 不是聊天记录。

不是 Prompt。

不是课程笔记。

不是知识摘抄。

Hermes Memory 是：

> **经过人工审核、交叉验证、长期有效、能够指导决策的专家认知。**

每一条 Memory 都应该能够：

* 被 AI 理解
* 被 AI 调用
* 被人维护
* 被长期更新
* 被版本控制

---

# 二、Memory 编号规范

统一格式：

```text
M0001
M0002
M0003
...
```

编号永不复用。

即使删除，也不重新编号。

例如：

```text
M0008（Deprecated）
```

而不是重新分配。

---

# 三、Memory 分类

Memory 分四层。

---

## L1 世界观（Worldview）

几乎永远不会改变。

例如：

* 创作哲学
* 用户本质
* 内容价值
* 长期主义

---

## L2 战略（Strategy）

几年变化一次。

例如：

* 起号
* 推荐机制
* IP成长
* 商业模式
* 用户心理

---

## L3 战术（Tactics）

实际操作。

例如：

* 昵称
* 简介
* 封面
* 标题
* 脚本
* 剪辑

---

## L4 平台（Platform）

平台规则。

变化最快。

例如：

* 抖音政策
* 推荐机制更新
* 新功能
* 新工具

# 十、最终原则

Hermes Memory 的目标不是收集知识。

> **沉淀真正属于专家的大脑。**

为什么不是其它方法？

## 我建议在 V0.2 再增加一个非常关键的设计

今天写完这份规范后，我意识到还有一个比 `Memory` 更小的知识单位。

我建议引入 **Memory Atom（记忆原子）** 的概念。

也就是说：

* **Memory Atom（MA）**：一句不可再拆分的专家认知，例如"封面的唯一目标是提高点击率"。
* **Memory（M）**：由多个 Memory Atom 组成的一篇完整专家记忆，例如《M0005_封面设计》。
* **Framework（F）**：调用多个 Memory 去完成分析。
* **Skill（S）**：调用 Framework、Memory、Checklist 去执行任务。

这样，未来你的知识系统会形成一棵真正的知识树，而不是一堆 Markdown 文件。例如：

```text
MA0001 → 封面的唯一目标是提高点击率
MA0002 → 一张封面只表达一个信息点
MA0003 → 封面文字控制在可快速识别的范围

        ↓

M0005《封面设计》

        ↓

F0003《封面诊断框架》

        ↓

S0008《封面优化 Skill》
```

我认为这是整个系统未来最值得加入的一项设计，因为它会让 Hermes 的知识真正具备**可组合、可复用、可推理**的能力，而不仅仅是读取整篇文档。
