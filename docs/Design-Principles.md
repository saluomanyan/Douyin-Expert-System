# Design Principles

> 本项目遵循的核心设计原则

---

## 第一原则：宁缺毋滥

> 宁可 Memory 为空，也不收录不合格的知识。

- 不追求数量（"800 条 Memory"），追求质量（每条都能独立支撑决策）
- 未通过 6 条铁律的，一律不进 Memory
- 怀疑的、不确定的，留在 `knowledge/` 待验证

---

## 第二原则：操作锚点

> 每一条 Memory 必须能直接指导一个具体动作。

- ❌ "做好定位" — 这是标题，不是知识
- ✅ "昵称不超过 2 个信息点，1 秒内可理解" — 这是操作锚点

---

## 第三原则：解释为什么

> 不仅告诉"做什么"，还揭示"为什么"。

- ❌ "前 3 秒要吸引人"
- ✅ "前 3 秒唯一目标是阻止划走。因为算法在第一波推荐中只看停留率。如果用户 3 秒内划走，系统判定内容不吸引人，不给第二波流量。"

---

## 第四原则：多源验证

> 每条 Memory 至少来自 3 个独立来源。

- 单一来源 = 个人经验 = 暂时不收录
- 多人共识 = 行业规律 = 值得收录

---

## 第五原则：跨时间复用

> 优先底层规律，不追热点技巧。

- 底层规律：为什么用户会关注？为什么评论区是第二内容池？
- 热点技巧：2026 年某流量扶持政策（放 `knowledge/`，由搜索获取）

---

## 第六原则：创作者哲学优先

> 任何运营建议不得损害长期 IP。

- 如果某个技巧 CTR 高但损害 IP → 拒绝
- 如果某个技巧短期涨粉但违背创作理念 → 拒绝

---

## 第七原则：框架无关

> 知识工程本身不绑定任何 Agent 框架。

- Hermes 是第一个目标平台，但不应该是唯一平台
- 通过 `adapter/` 实现框架接入
- 知识本身保持纯 Markdown，可被任何系统读取
