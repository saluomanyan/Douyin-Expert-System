# Design Principles · 设计原则

---

## 原则一：先知识工程，再 Agent 工程

> 第一版不要考虑 Hermes 适配。先做知识工程，再做 Agent 工程。

- 这个 GitHub 项目本身是一套完整的「抖音专家知识库」
- 知识工程与 Agent 框架完全解耦
- 以后 Hermes、LangGraph、OpenManus、Claude Code 都能加载

---

## 原则二：知识独立于 LLM

> LLM 只是大脑，知识库才是核心竞争力。

- 不信任 LLM 自带的训练知识（来源杂、质量不齐、可能过时）
- Memory 完全独立于 LLM
- 换 DeepSeek / Qwen / Claude / GPT / Gemini，认知不变

---

## 原则三：写"为什么"，不写"怎么做"

> 不是教程，是认知。不是 Prompt，是 Memory。

- ❌ "写一个吸引人的标题" — 这是操作指令
- ✅ "标题不是介绍视频，而是制造点击理由" — 这是认知
- 每条 Memory 必须解释底层机制

---

## 原则四：交叉验证

> 只有一个人在说的知识，暂时不收录。

- 每条核心认知至少需要 3 个独立来源验证
- 来源标注遵循 Citation-Standard

---

## 原则五：人工最终审核

> AI 负责搜集和提炼，人负责最终筛选。

工作流程：
1. AI 全网寻找、提炼、交叉验证
2. 人阅读、修改、删减、补充
3. 最终进入 Hermes Memory

---

## 原则六：以 Commit 为推进单位

> 不是积累聊天记录，而是持续构建一个知识系统。

- 每个 Phase / Stage 对应一次 GitHub Commit
- 可追踪、可回滚、可审核

---

## 原则七：高密度、可演化

- 不是经验故事，而是可复用原则（跨赛道迁移）
- 不是平台规则复述，而是底层机制解释
- 不是碎片技巧，而是能组成完整决策体系
